## 2. 数据源与口径

### 2.1 系统成交量来源

`etf_v7_threefactor.py` 使用：

- 主数据源：腾讯财经 K 线 HTTP API  
  `http://web.ifzq.gtimg.cn/appstock/app/fqkline/get?param={pfx}{code},day,,,{limit},qfq`
- 解析字段：`data.{pfx}{code}.day` 或 `data.{pfx}{code}.qfqday`
- 每行 K 线字段：`[date, open, close, high, low, volume]`
- 成交量字段：`r[5]`，脚本直接读取为 `float(r[5])`
- 备用：新浪实时行情，仅在腾讯失败时补当日单条

本系统没有使用 akshare 作为主成交量来源。东方财富实时 API 的 `f84` 用于 ETF 流通份额/份额变化因子，不是本次成交量核验的主口径。

### 2.2 量能因子口径

系统实际没有计算统计意义上的“成交量分位数”。它计算的是：

```text
vr = 当日成交量 / 前20个交易日成交量均值
vp = vprob(vr)
```

其中 `vprob()` 是分段线性映射。综合分中：

```text
份额因子可用：cp = vp * 0.5 + dp * 0.2 + sp * 0.3
份额因子不可用：cp = vp * 0.7 + dp * 0.3
```

## 4. 异常性判定

本次采用以下阈值：

- `< 1.0x`：低于均量，不是放量
- `1.0x ~ 1.5x`：接近/略高于均量，不构成异常
- `> 1.5x`：温和放量
- `> 2.0x`：明显放量
- `> 3.0x`：极端放量

四只沪深300 ETF 的目标日成交量倍率全部低于 `0.5x`。因此，按系统口径，2026-06-10 沪深300 ETF 没有成交量异常放大。

## 5. 三因子系统为何未检测到异常

三因子系统未报异常的直接原因是量能因子非常低：

- `510300`: `vr=0.455876`, `vp≈4.6`
- `510310`: `vr=0.240076`, `vp≈2.4`
- `510330`: `vr=0.274781`, `vp≈2.7`
- `159919`: `vr=0.247134`, `vp≈2.5`

在份额因子可用时，量能权重为 50%，但因为 `vp` 本身极低，量能部分对综合分贡献只有约 `1.2~2.3` 分，不可能推动综合分进入异常区间。

## 6. Bug 结论

未发现量能因子计算 bug。

已用两种方式对照验证：

1. `verify_volume.py` 直接拉腾讯 K 线，计算目标日成交量 / 前20日均量。
2. 调用 `etf_v7_threefactor.analyze_all()` 对同一批 K 线数据计算 `v/vma/vr/vp`。

两者结果一致，仅存在四舍五入差异。

### 口径/文案风险

如果系统界面或说明称“量能分位数”，容易让人理解为统计 percentile；实际实现是“相对前20日均量倍率 + 分段映射”。建议后续将文案改为“量能倍率”或“20日均量倍率映射分”，避免误解。

## 7. 后续建议

如果用户仍观察到“2026-06-10 沪深300 ETF 异常”，建议核对是否使用了不同口径：

1. **成交额而非成交量**：价格变化或成交额口径可能与 K 线 volume 不同。
2. **盘中实时盘口**：盘中瞬时放量不一定在日 K 成交量上表现为异常。
3. **ETF 范围不同**：是否包含更多沪深300相关 ETF、联接基金、增强 ETF，或只看了某一只非上述代码。
4. **数据源不同**：交易所/东方财富/同花顺/雪球等可能展示单位不同，需统一到同一字段和单位。
5. **目标日期时区/交易日误差**：确认“昨日”是否确指 `2026-06-10`，而不是 `2026-06-11` 盘中或其他本地时区日期。
